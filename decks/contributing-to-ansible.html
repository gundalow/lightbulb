Development Process: Aims of reviews

  • We need confidence before code can be merged
  • We need user feedback to tell us if this is right
  • One of the main reasons for the PR backlog size is due to lack of review

Development Process: How to review

  • Reviewing PRs is one of the most beneficial things to do
  • Three part review:
    1. Is the idea behind the contribution sound? - Clear, consistent user interface
    2. Is the contribution architected correctly? - non-user facing
    3. Is the contribution polished? - Docs, Tests, etc
  • Be mindful
    • Go above and beyond to promote a collaborative and respectful community
    • "Would this be clearer if..." - invites discussion
    • Avoid bikeshedding. Ansible's coding guidelines is enforced by CI.
  • The Gentle Art of Patch Review
  • We know the "how to review" parts of Ansible's Docs need some work, what would you like to see there? - give comments in Chat or join the discussion in #ansible-community

Development Process: Types of review

  • Functional review
    • Does it work?
    • Address a use case you have?
    • Does it break and edge/use case you have?
    • Is the interface clear
  • Documentation reviews
    • Are the docs clear, is there any ambiguity?
    • For modules, are the examples clear and useful?
  • Code reviews
    • Does the PR make the best use of APIs or optimization or indicating usefulness of code?
    • Does the PR follow Ansible's coding guidelines?
    • Does it follow the CRUD principles (Create, Read, Update, Delete)?
  • One thing I think is often overlooked here is that you don't need to be a programer, or have any understanding of the internals to do a functional or documentation review.
  • We know the "how to review" parts of Ansible's Docs need some work, what would you like to see there? - give comments in Chat or join in #ansible-community
  • I accept some of this is subjective, so we are looking for a net improvement
