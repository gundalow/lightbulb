Myths

  • Attending events, 200+ Meetups, regular Ansible Automates & Webinars
  • Join IRC/Mailing Lists
  • MYTH: I need to be an expert in Python to contribute
    REALITY: You don't even need to know Python to review PRs, contribute docs or test
  • MYTH: Writing tests is difficult
    REALITY: if you can write a Playbook you can create, or extend, an integration test
  • MYTH: I'm brand new to Ansible, I can't contribute
    REALITY: You can learn and contribute
  • MYTH: I need to know Git
    REALITY: You can fix issues right in GitHub. Reviewing and commenting is just as important.
  • MYTH: I need to make a commitment to stay around and support
    REALITY: A drive-by contribution that clarifies documentation, fixes a bug, or adds a test case is always welcome. Though adding new modules we look for some ongoing support

GitHub Issues and PRs

  • How everything is tracked
  • GitHub Issues/PRs or it never happened
  • Issues
    • Used for bug reports and feature ideas
    • Can be used to discuss ideas before implementation starts
  • Pull Requests (PRs)
    • Improve a single thing: Fix a bug or add a feature
    • Always use feature branches, never work directly on devel
    • Used for bug reports and feature ideas
  • If you know the fix, just create a PR, no need to create an issue first
  • New to GitHub: lab.github.com, guides.github.com

Development Process: Aims of reviews

  • We need confidence before code can be merged
  • We need user feedback to tell us if this is right
  • One of the main reasons for the PR backlog size is due to lack of review

Development Process: Types of review

Whatever your skill set, you can help with at least one of these:

  1. Functional review
  2. Documentation reviews
  3. Code reviews

Development Process: How to review

  • Reviewing PRs is one of the most beneficial things to do
  • Three part review:
    1. Is the idea behind the contribution sound? - Clear, consistent user interface
    2. Is the contribution architected correctly? - non-user facing
    3. Is the contribution polished? - Docs, Tests, etc
  • Be mindful
    • Go above and beyond to promote a collaborative and respectful community
    • "Would this be clearer if..." - invites discussion
    • Avoid bikeshedding. Ansible's coding guidelines is enforced by CI.
  • The Gentle Art of Patch Review
  • We know the "how to review" parts of Ansible's Docs need some work, what would you like to see there? - give comments in Chat or join the discussion in #ansible-community
